Part One: Scratching My Head.

My initial plan was to create an application where a user could upload a photo of a bookshelf full of books. My app would then use image recognition and machine learning to identify the specific edition of each and every book in the photo. It would then automatically add all of those books to the user's database.

However, I didn't even make it past "Part One: Scratching my head..." before I realized that I should probably scale back the idea a bit seeing as this is my first proper website. So I decided not to abandon my initial idea, but to instead build the application in stages. Stage one, the goal of this project, would be to create a website where users can search for books and add them to their bookcases. Simple as that, or so I thought.

At this point three months into my learning journey, I have a basic understanding of HTML, CSS, Python, SQL, and JavaScript. I've also started using the basic functionality of Git and GitHub. I learned the Flask basics in Week 9 of CS50x where I completed the infamous "Finance" problem set. Despite that, I felt overwhelmed by the thought of creating my own Flask application from scratch. I took a step back and followed along with some tutorials. I also actually read the documentation for Flask and Jinja. This was a bit of a milestone in that previously I'd try to read the documentation for a library or framework and it would be like reading a foreign language. This time, I was able to understand the documentation and use it to my advantage.

By far the most helpful tutorials I found were by Youtubers @bitfumes and @TechWithTim. I greatly appreciate their help in getting many concepts to finally start clicking for me.

Armed with a better understanding of Flask, I was able to start building my application. I created my repository "bookcase-database" and got to work thinking about the bones of the project, its basic file structure.

Part Two: Enter the Matrix

(aka WTF is a virtual environment)

As soon as I started reading about creating web applications with Flask, I kept seeing the term "virtual environment" pop up. I had no idea what this was, but I knew I needed to figure it out. I did some reading and asked ChatGPT some questions.

I learned that a virtual environment is a way to isolate your project from the rest of your computer. This is useful because it allows you to install packages and libraries specific to your project without affecting the rest of your computer. Once I understood and figured out how to create a venv and activate it, I found it incredible useful. I appreciate being able to view all the packages I've installed for a project and know they're not some unrelated package installed on my computer that I don't understand the purpose of (there's still a lot of those). So for me with this first project at least, the real value of using a venv was the ability to use "pip list" and "pip freeze > requirements.txt". This formed the beginning of my understanding of dependencies and requirements. Much later in the project, I learned that virtual environments are also useful for deploying your application.

I was able to get the venv up and running on the first day of working on the project. I think the highlight of this part was when I created my first bash script that actually proved useful. All it did was save me a few seconds of typing "source venv/bin/activate" but it was a proud moment for me. This script would evolve a bit over the course of the project to include exporting environment variables (which was another foreign concept to me at this point).

Part Three: Put the db in bc.db

Actually, I think it's pronounced "Skewl"

The next step was creating a database. I decided to use SQLite for this project. I had used it before in CS50x and it seemed like a good choice for a small project like this. I also decided to use SQLAlchemy as an ORM. I didn't know what that was until I started preparing for this project. I learned that an ORM is an Object Relational Mapper. It's a tool that allows you to interact with a database using objects and methods instead of SQL queries. I decided to use an ORM because I thought it would be a good learning experience. I also thought it would be useful to know how to use an ORM for future projects.

I loved the SQL week of CS50x and I was excited to get to use SQL again. I knew my database would need a table for users, a table for books, and a table for bookcases. I connected the bookcase table to the user table through the "owner_id". This means that each bookcase would belong to one user. A user can have many bookcases, but each bookcase can only belong to one user. I used a many-to-many relationship between the bookcase table and the book table. This means that a bookcase can have many books and a book can be in many bookcases. I used a join table called "bookcase_book" to connect the two tables. This table has a bookcase_id and a book_id.

I used the "flask db migrate" command to create a migration file. I then used the "flask db upgrade" command to apply the migration to my database. I was able to use the "flask shell" command to test out my database. I was able to create a user, bookcase, and book and add the book to the bookcase. I was also able to query the database to get the books in a bookcase and the bookcases a user owns. I used the "flask db downgrade" command to undo the migration and start over. I did this a few times as I made changes to my database. Though much of the time, I just opened a second terminal and used the "sqlite3" command to open the database and make changes that way.

In order to connect the database to my flask project, I created a "models.py" file. I imported SQLAlchemy and created a db object. I then imported the db object into my "app.py" file and initialized it. I also imported the db object into my "auth.py" file so I could begin the next stage of the project: user authentication (dun dun dun).

Part Four: Inside Out

Register, Login, and Logout

Before I started this part of the project, I used Jinja to create a base.html file and a home.html file. I used the "extends" and "block" keywords to create a base template that I could extend to other pages. I also created a "static" folder for CSS and JavaScript files, and a "templates" folder for all the HTML files.

Thanks to TechWithTim's Flask tutorials, I was able to get user authentication up and running pretty quickly. I created a "auth.py" file and imported the db object from my "models.py" file. I created a User class and a Bookcase class. I used the UserMixin class from Flask-Login to add some useful methods to my User class. I created a login_manager object and initialized it in create_app() function inside my __init__.py file. I created a login.html file and a register.html file, along with a login route and a register route. I used the "flask-login" extension to create a login_required decorator for my routes and finally created a logout route.

Flask-Login was fairly straight forward to use. As I type this I'm trying to think of issues I had setting this up, but I can't think of any. I think the only part I remember struggling to understand was the "current_user" variable. But turns out it's just a variable that stores the current user...duh. I guess I just expected more steps to make that work.

Oh! I almost forgot, password hashing! I used the "werkzeug.security" module to hash passwords. I used the "generate_password_hash" function to hash the password when a user registers. I then used the "check_password_hash" function to check the password when a user logs in. Easy peasy.

Part Six: The Keys to the Kingdom

How to use API's

I knew I wanted to use the Google Books API to get book information for my application. I had no idea how to use an API or how to get an API key. I did some reading and watched some videos and I was able to get a basic understanding of how API's work. I learned that an API is a set of functions and procedures that allow the creation of applications that access data and features of another service. I learned that an API key is a unique identifier that's used to authenticate a user, developer, or program to an API. I learned that API keys are used to track and control how the API is being used, for example to prevent malicious use or abuse of the API.

This is also the part where I had to learn about environment variables. I learned that environment variables are variables that are set in the operating system or in the terminal. They store information that is used by applications. When I was developing locally, I just used a bash script to activate my venv and export my environment variables. However, later before I deployed the website, I ended up using the "python-dotenv" module to create a .env file and store my environment variables.

To start with, I created a search form with three fields -- title, author, and isbn. I created a route for the search form and a view to handle the form data. I used the requests module to make a GET request to the Google Books API. The hardest part was figuring out how to format the search query itself. I ended up using "f-strings" to format the query and .join() to join each word in the query with a "+". Once I got the search results back, I was able to display them on the search page along with a thumbnail image of each book's cover. I also added a button to each book that would allow the user to add the book to their bookcase.

But then I ran into a problem. How do I actually add the book to the database? Seeing as I already had a working form to manually add books, I realized all I had to do was create an invisible form for each book in the search results and use those hidden inputs to pass the data to the /add_book route.

Part Eight: The Home Stretch

Testing, Debugging, and Missing Features

I was nearing the end of the project and I was feeling pretty good about it. I had a working application that I was proud of. I had learned a lot and I was excited to deploy it. But I knew I had to test it first.

I started by testing the routes and views. I went through each page and made sure everything was working as expected. I made a list of all the features I wanted to test and checked them off the list one by one. I ran into problems immediately.

First, I decided to make the username case-insensitive so that "JohnDoe" and "johndoe" would be the same username, while the database remembers the capitalization that the user used when they registered. I also adjusted the login route code to allow for the user to enter their Email Address or username.

The next problem that I encountered was on the /search page. My application displays a list of all the books that match the user's search query. Each book has a button that allows the user to add the book to their bookcase. I noticed that if the user clicked the button to add the book to their bookcase, the book would be added to the database, but the page would refresh and the search results would disappear. I decided to figure out a way to refresh the page with the same search results after the user added a book. I was able to accomplish this by passing the search query to the /add_book route and then passing the search query back to the /search route with render_template().

It was during testing that my sloppy CSS came back to bite me a few times. I spent too much time trying to figure out why one button wasn't styled correctly, or one form didn't have the right box shadow, etc. This just reinforced the need for me to learn more about CSS and how to keep things organized.

On the edit_book.html page, I have a form with inputs and textareas for all the forward facing book information. For the "user rating" field I used a select element with options for 1-5 stars and a checkbox for "read". It took me a while to finally get this working as intended. I had to figure out how to pass the current user rating and read status to the form. I ended up more hidden input fields to actually hold the necessary values then used JavaScript to pass these values to the correct fields.

Part Nine: Forgot Password?

Flask_Mail, itsdangerous, tokens, secrets, SMTP, serialized tokens...ahhh

I added a "Forgot Password?" link to the login page. I decided to break this task up into two parts. First I would figure out how to use Flask-Mail to send an email to the user with a link to a page where they could change their password. Then I would figure out how to make this link secure.

Configuring Flask-Mail was fairly straight forward. I initially created a Mail object and configured mail within my app.py file. But because of my project's file structure, I ended up creating the object within the __init__.py file's create_app() function. I also added another two environment variables to my .env file, one for the gmail username and one for the password. I used the "os" module once again to access these environment variables.

The next step wasn't quite as simple. I started by installing the Python "secrets" module and the "itsdangerous" library. I used secrets to create a secret key for the token and used itsdangerous to create a token that would be sent to the user's email address. I spent an inordinate amount of time trying to discern the differences between 'TimedJSONWebSignatureSerializer' and 'URLSafeTimedSerializer', and figuring out how it all actually works. My biggest issue was trying to set the expiration time within the parameters of the URLSafeTimedSerializer object. I must have come across some post or maybe copilot (yea you) suggested I use "expires_in" within the parameters of the object when I was creating it. But turns out, I had to use "max_age" as an argument for the loads() method when I was decoding/authenticating the token. I enjoyed troubleshooting this bug as it was one of the first times I felt comfortable following along with a more "advanced" topic in Stack Overflow threads.

It was a genuinely great feeling when I heard my phone ding right after testing the reset password functionality and saw an email from "BookcaseDatabase@gmail.com" with a link to reset my password. I clicked the link, entered a new password, and was redirected to the login page. The new password worked!

Part Ten: ...9...8...7...6...

Deployment!!! (and a few more bugs)

Throughout my time working on the project there were a few moments where I stopped and wondered how certain parts of the project would work when I deployed it. I knew I would have to figure out how to use environment variables outside of running my little bash script each time I activated the virtual environment. I'd have to figure out how to use a different database and how to connect it to my application. But to start with I had to be able to answer the question -- "How do I deploy a Flask application?"

Right off the bat, I kept seeing the same few options repeated over and over. PythonAnywhere, Heroku, Docker and AWS. But first I decided to finally figure out what "WSGI" meant. I knew that Flask was defined as a "WSGI" framework, but I had no idea what that meant. One quick google search later and I had my answer -- "Web Server Gateway Interface". I learned that WSGI is a specification for a universal interface between web servers and web apps. In other words, it's a way for a web server to communicate with a web application.

I flipped a four sided coin and decided to go with PythonAnywhere. I followed step by step instructions on their website and was able to get my site up and running fairly quickly. I used the "git clone" command to clone my project from GitHub directly into PythonAnywhere. The only thing that gave me a little trouble was setting up and using python-dotenv and getting the virtual environment within PythonAnywhere to include the dependencies I needed. I had some trouble getting pipreqs to work, but I found a post on Stack Overflow where the author answered their own question. Turns out that because my .venv/ folder is in my main directory, pipreqs was looking for all the dependencies within that folder which was causing it to generate a weird requirements.txt file. I fixed this by using the pipreqs option '--ignore' when executing the command.

I shouldn't have been so stressed about the deployment process. It ended up being very straight forward. The only other issue that I encountered had to do with Flask Mail (yet again). I mentioned this in the previous section, but this is when I actually had to move my initialization of the Mail object from my app.py file to my __init__.py file. I also had to change the way I was importing the Mail object in my auth.py file. This was the first time I actually began to feel like I understand how Python imports work.
